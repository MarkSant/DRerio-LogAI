

## Relatório de Auditoria Técnica — ZebTrack-AI

**Escopo**: Análise exaustiva do repositório, cobrindo arquitetura, qualidade, performance, segurança e roadmap.
**Postura**: Independente. Verdade técnica sem amenidades.

---

### 1. Arquitetura e Design

**Padrão declarado**: MVVM-S (Model-View-ViewModel-Service) com Dependency Injection manual e Event-Driven Architecture.

**Estrutura física**: A separação em camadas é clara e bem demarcada:

| Camada | Pacote | Responsabilidade |
|--------|--------|-----------------|
| Presentation | `ui/` (gui, dialogs, wizard, components, builders) | Tkinter GUI |
| ViewModel | `core/main_view_model.py`, `core/viewmodels/` | Orquestração |
| Coordinators | `coordinators/` (10 coordinators) | Workflows complexos |
| Services | `core/*_service.py`, `core/*_manager.py` | Lógica de domínio |
| I/O | `io/` (recorder, video_source, camera, arduino) | Persistência |
| Analysis | `analysis/` (behavior, reporter, roi, data_transformer) | Computação científica |
| Plugins | `plugins/` (base, ultralytics, openvino) | Strategy Pattern para detecção |

**Composition Root** (\_\_main\_\_.py): DI manual, com todas as dependências criadas e injetadas em ~350 linhas. Uso de dataclass `MainViewModelDependencies` como *parameter object* para 22+ campos — padrão adequado para evitar construtores com dezenas de parâmetros.

**O que funciona bem (SOLID)**:
- **Single Responsibility** — camadas `io/`, `plugins/`, `analysis/`, `tracker/` são coesas e auto-contidas
- **Open/Closed** — o sistema de plugins usa Strategy via `DetectorPlugin` ABC. Adicionar um novo backend (TensorRT, ONNX) não exige alterar o código existente
- **Dependency Inversion** — `settings_obj` é injetado via construtor, nunca importado como singleton. `TYPE_CHECKING` imports quebram ciclos de dependência
- **Interface Segregation** — exceptions.py define 22 exceções granulares em hierarquia domain-specific

**Falhas arquiteturais graves**:

1. **God Classes endêmicas** — processing_coordinator.py com **5.433 linhas e 113 métodos** viola brutalmente o SRP. Concentra processamento de vídeo, zona, multi-aquário, sequenciamento, análise, e geração de relatórios. project_manager.py com **3.227 linhas/95 métodos** é o segundo pior.

2. **Dois `BaseCoordinator` coexistindo** — base.py (ABC, 305 linhas) e base_coordinator.py (concreto, 136 linhas). Diferentes coordinators herdam de bases diferentes, criando inconsistência estrutural inexplicável.

3. **Dois EventBus coexistindo** — `event_bus.py` (v1, queue-based, 273 linhas) e `event_bus_v2.py` (pub/sub). Nem a migração foi concluída nem o antigo foi removido, o que cria ambiguidade sobre qual usar.

4. **Inicialização em duas fases** — `MainViewModel` usa pattern `__new__` + proxy no `__main__.py` para contornar dependências circulares entre ViewModel ↔ GUI. Isso é um *workaround* que sinaliza acoplamento estrutural não resolvido.

5. **`core/` é um *dumping ground*** — contém 40+ arquivos misturando services, managers, facades, viewmodels, coordinators, schemas, types, e um bootstrapper. Falta sub-pacotização por domínio.

6. **`orchestrators/` legado** — pacote com 2 ficheiros (`ui_state_controller.py`, `video_processing_orchestrator.py`) que deveriam ter sido migrados para `coordinators/`. `ApplicationBootstrapper` ainda importa e instancia coordenadores legados.

**Nota Arquitetura: 6.5/10** — Fundação sólida; execução incompleta. A visão é clara (MVVM-S + DI + Events), mas migrações estagnadas e concentração de responsabilidades destroem a manutenibilidade.

### 2. Qualidade do Código e Legibilidade

**Convenções de nomenclatura**: Consistentes — `snake_case` para funções/variáveis, `PascalCase` para classes, `UPPER_SNAKE_CASE` para constantes, `_` leading para membros privados. Aderência ao PEP 8 nos 100% dos ficheiros verificados. **Nota: 8/10**.

**Type hints**: Extensivas. Praticamente todas as funções públicas são tipadas. 115 `# type: ignore` existem, concentrados em settings.py (16), \_\_main\_\_.py (14), e ultralytics_detector.py (12). Indica fricção com tipagem que deveria ser resolvida, não suprimida.

**Docstrings**: Qualidade acima da média. Classes e funções públicas usam Google-style docstrings com `Args`, `Returns`, e `Examples`. Problema: **idioma inconsistente** — `gui.py` usa português ("Este módulo define..."), enquanto `coordinators/` usa inglês.

**God Files** (> 1.000 linhas):

| Arquivo | Linhas | Métodos | Severidade |
|---------|-------:|--------:|------------|
| processing_coordinator.py | 5.433 | 113 | CRÍTICA |
| project_manager.py | 3.227 | 95 | CRÍTICA |
| detector.py | 2.605 | — | ALTA |
| gui.py | 2.250 | 143 | ALTA |
| canvas_manager.py | 2.152 | — | ALTA |

**31 ficheiros** excedem 500 linhas. São 5 ficheiros acima de 2.000 linhas.

**Code Smells detectados**:



| Smell | Contagem | Impacto |
|-------|-------:|---------|
| `except ... : pass` (exceções engolidas) | **94** | Mascara bugs em produção |

| Placeholder URL não atualizada | 1 | menu_manager.py: `"https://github.com/YOUR_USERNAME/ZebTrack-AI"` |

**Ficheiro mais perigoso**: state_synchronizer.py — **14 instâncias** de `except Exception: pass`. A camada de sincronização de estado UI silencia todas as falhas, tornando bugs de estado impossíveis de diagnosticar.

**Nota Qualidade: 5/10** — Nomenclatura e docstrings são fortes, mas a concentração massiva em god classes e o volume de exceções engolidas são incompatíveis com um software maduro.

### 3. Análise de Funções e Lógica

**Pipeline de Processamento Principal**:

```
VideoCapture.read() → Detector.detect() → Recorder.write_detection_data() → AnalysisService → Reporter
```

O `ProcessingWorker` usa `multiprocessing.Process` — **contorna o GIL completamente** para processamento de vídeo. Design correto.

**Otimizações efetivas já implementadas**:
- **Frame skipping** com `cap.grab()` (10-20x mais rápido que `cap.read()`) para frames não processados
- **FP16 half-precision** auto-habilitado em GPU CUDA
- **OpenVINO**: performance hints (`LATENCY`/`BATCH`), `decode_masks=False`, `CACHE_DIR`, reuso de infer request
- **RecorderFactory**: lazy-loading com double-checked locking — economiza ~2.9s no startup e ~150MB de memória
- **Streaming Parquet**: dual trigger (500 rows OR 5 segundos), compressão Snappy
- **Análise comportamental**: `calculate_total_distance()`, `detect_freezing_episodes()`, `get_velocity_stats()` usam NumPy vetorizado — O(n) ✅

**Gargalos de performance identificados**:

| # | Gargalo | Impacto | Local |
|---|---------|---------|-------|
| 1 | **Modelo de segmentação usado só para detecção** | ALTO — 30-50% de desperdício computacional. O modelo `*_seg.pt` computa masks que são descartadas. | detector.py |
| 2 | **Batch inference existe mas não é usado** | MÉDIO-ALTO — `detect_batch()` faz fallback para sequential por frame. | detector.py |
| 3 | **Preview frames serializados via Queue** | MÉDIO — Frames BGR completos (~3-5MB cada) passam por pickle no multiprocessing Queue | processing_worker.py |
| 4 | **Angular velocity com Python loop** | MÉDIO — `for i in range(...)` com `np.arctan2` por ponto. Deveria ser totalmente vetorizado | behavior.py |
| 5 | **Parquet flush via DataFrame(list-of-dicts)** | BAIXO-MÉDIO — Conversão dict→DataFrame a cada 500 rows. Buffers colunares seriam mais eficientes | recorder.py |

**Nota Performance: 7.5/10** — O pipeline principal é bem arquitetado (multiprocessing, grab(), streaming parquet). Os gargalos são otimizações de segundo nível, exceto o modelo seg usado para detecção que é desperdício significativo.

### 4. Segurança e Tratamento de Erros

**Postura geral de segurança**: **Boa para uma aplicação desktop de pesquisa**.

**Vulnerabilidades encontradas**:

| Severidade | Achado | Local |
|------------|--------|-------|
| **MÉDIA** | `pickle.load()` em cache local — permite execução arbitrária se diretório comprometido | metrics_cache.py |
| **MÉDIA** | `Recorder.detection_data` (lista partilhada) sem lock entre threads | recorder.py |
| **BAIXA** | `subprocess` para `os.startfile`/`xdg-open` — todos com paths gerados internamente, não de input do utilizador | Múltiplos ficheiros |
| **NENHUMA** | Sem `eval()`, `exec()`, bare `except:`, credenciais hardcoded, YAML unsafe | — |

**Elementos de segurança bem implementados**:
- **Pydantic v2 com `extra="forbid"`** — rejeita campos desconhecidos na configuração
- **`yaml.safe_load()`** em todos 4 sites de parse YAML
- **Path traversal checks** em wizard/models.py — valida contra diretórios de sistema
- **Path sanitization** em `project_service.py` — strip de caracteres especiais
- **.gitignore robusto** — exclui config.local.yaml, `.env`, `*.local.yaml`
- **Logs JSON via structlog** — sem dados sensíveis nos logs
- **Hierarquia de exceções completa** — 22 classes domain-specific em exceptions.py

**Tratamento de erros — análise detalhada**:

| Padrão | Avaliação |
|--------|-----------|
| Caminhos críticos (`recorder.py`, `video_source.py`, `detector_service.py`) | **Excelente** — exceções específicas (`ValueError`, `FileNotFoundError`, `IntegrityError`), schema validation imediata |
| Coordinators/Services | **Bom** — `except Exception as e` com structlog logging, keys domain-specific |
| UI/State Synchronizer | **Ruim** — 94 `except…pass` silenciam falhas. `state_synchronizer.py` é o pior ofensor |
| Threading | **Bom** — `RLock` em state_manager/project_manager, `daemon=True` em todas threads, observer timeout 5s |

**Lacuna de segurança em CI**: Nenhum `pip-audit`, `safety`, ou `snyk` configurado. Dependências não são scanned automaticamente.

**Nota Segurança: 7/10** — Sem vulnerabilidades críticas. Os pontos de atenção (pickle, recorder lock, exceções silenciadas) são mitigáveis com esforço moderado.

### 5. Pontos Fortes e Críticas Construtivas

#### O que foi bem implementado

1. **Composition Root com DI manual** — Abordagem disciplinada; `MainViewModelDependencies` como parameter object é limpo; settings nunca importado como singleton
2. **Plugin system** — Strategy pattern via `DetectorPlugin` ABC; suporte YOLO + OpenVINO com fallback gracioso; extensível sem alterar código existente
3. **Hierarquia de exceções** — 22 classes específicas por domínio, com docstrings, cobrindo todos os subsistemas
4. **Configuração Pydantic v2** — Layered config (config.yaml < config.local.yaml), `extra="forbid"`, validadores custom (divisibilidade, existência de paths)
5. **RecorderFactory** — Lazy loading com double-checked locking, proxy transparente, economia real mensurável (2.9s + 150MB)
6. **Suite de testes robusta** — 2.568 testes, marcadores `gui/slow/integration`, fixtures sofisticadas, `pytest-timeout`, hook `sessionfinish` para cleanup forçado
7. **StateManager thread-safe** — Observable state com `RLock`, ThreadPoolExecutor para notificar observers com timeout, histórico de estados
8. **Pipeline multiprocessing** — Processamento em processo OS separado, `cap.grab()` para frame skip, streaming Parquet

#### O que é medíocre e exige refatoração

1. **`ProcessingCoordinator` é insustentável** (5.433 linhas) — Nenhum engenheiro consegue revisar, testar isoladamente, ou modificar com confiança um ficheiro com 113 métodos. O Phase 3 de extração *moveu* o problema em vez de resolvê-lo.

2. **94 exceções silenciosas são uma bomba-relógio** — Particularmente em `state_synchronizer.py` (14x), estas mascararam bugs em produção. Quando o estado da UI corrompe silenciosamente, o diagnóstico é impossível.

3. **Migração EventBus v1→v2 estagnada** — Duas implementações coexistindo cria incerteza para novos contribuidores sobre qual usar. Sem timeline de depreciação.

4. **Dois `BaseCoordinator` é negligência técnica** — base.py (ABC) e base_coordinator.py (concreto) fazem a mesma coisa com interfaces diferentes. Alguns coordinators herdam de um, outros do outro.

5. **115 `type: ignore` suprimem problemas reais** — Em `settings.py` e `__main__.py` indicam que o type system não está sendo colaborado, mas combatido.

6. **Coverage gates modestos** — 55% Linux / 25% Windows são baixos para um software científico onde correção é crítica. GUI tests rodam sequencialmente (~7 min), criando atrito.

---

### 6. Sugestões de Melhoria (Roadmap) — 5 Ações Prioritárias

#### 1. **Decompor `ProcessingCoordinator`** (Prioridade: CRÍTICA)

Extrair em 4-5 coordinators focados: `VideoProcessingCoordinator`, `MultiAquariumCoordinator`, `SequentialProcessingCoordinator`, `ReportGenerationCoordinator`, `ProgressTrackingCoordinator`. Target: <800 linhas cada. Mesmo padrão já aplicado com sucesso na extração de dialogs de `gui.py` (de 10.759 para 2.250 linhas).

**Impacto**: Reduz risco de regressão, permite code review efetivo, melhora testabilidade.

#### 2. **Auditar e eliminar exceções silenciosas** (Prioridade: ALTA)

Revisar todas as 94 instâncias de `except…pass`. Ação mínima: substituir `pass` por `logger.debug("...", exc_info=True)`. Começar por `state_synchronizer.py` (14 instâncias) — esta camada está ativamente escondendo bugs. Depois `gui.py`, `window_utils.py`, `widget_factory.py`.

**Impacto**: Elimina a classe inteira de bugs silenciosos; reduz tempo de debugging drasticamente.

#### 3. **Unificar `BaseCoordinator` e concluir migração de legado** (Prioridade: ALTA)

Manter **apenas** base_coordinator.py (concreto, mais maduro). Migrar os poucos coordinators que herdam de base.py (ABC). Remover `orchestrators/` legado e os campos `DEPRECATED` em `MainViewModelDependencies`.

**Impacto**: Elimina confusão estrutural; reduz cognitive load para novos contribuidores.

#### 4. **Adicionar segurança de dependências em CI e substituir pickle** (Prioridade: MÉDIA)

Adicionar `pip-audit` ao pre-commit ou CI. Substituir `pickle.load` em metrics_cache.py por JSON ou MessagePack. Adicionar `threading.Lock` ao `Recorder.detection_data`.

**Impacto**: Fecha os 3 achados de segurança de severidade média simultaneamente.

#### 5. **Migrar modelo de segmentação para detecção-only e implementar batch inference** (Prioridade: MÉDIA)

O uso de `*_seg.pt` para tracking-only desperdiça 30-50% de compute. Treinar/converter um modelo detection-only. Simultaneamente, implementar batch inference real nos plugins para análise offline (2-4x throughput potencial).

**Impacto**: Ganho direto de performance sem alterar arquitetura; reduz custo computacional proporcionalmente.
